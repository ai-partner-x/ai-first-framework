## 测试框架概述

本项目使用 **Vitest** 作为单元测试框架，它是一个由 Vite 驱动的下一代测试框架，具有以下特点：

- 极速的热模块替换（HMR）支持，代码修改后立即重新运行测试
- 开箱即用的 TypeScript 支持，无需额外配置
- 与 Jest API 兼容，易于上手
- 内置代码覆盖率支持

### 为什么需要单元测试？

单元测试的主要目的是：

1. **保障代码质量** - 在修改代码后快速验证功能是否正常
2. **防止回归** - 确保新修改不会破坏已有功能
3. **文档作用** - 测试用例本身就是最好的代码文档
4. **提升信心** - 提交代码时更有底气

## 测试运行命令

### 项目测试目录说明

本项目有两个测试目录：

| 目录 | 配置文件 | 说明 |
|------|----------|------|
| `test/` | `vitest.config.ts` | 基础测试（service 层 + utils） |
| `test1/` | `test1/vitest.config.ts` | 扩展测试（controller + service + entity + dto） |

### 安装依赖

测试依赖已添加到 package.json，首次运行前需要安装：

```bash
cd scaffold/packages/api
pnpm install
```

### 运行测试

#### 1. 基础测试（test 目录）

```bash
pnpm test
```

**依赖配置**：
- 配置文件：`vitest.config.ts`（默认）
- 测试目录：`test/**/*.test.ts`
- 覆盖范围：service 层、utils 工具类

**执行过程**：
- Vitest 使用默认配置 `vitest.config.ts`
- 扫描 `test/` 目录下的测试文件
- 运行所有测试用例
- 输出测试结果统计（通过/失败数量）
- 然后自动退出

**适用场景**：日常开发快速验证、CI/CD 流水线

---

#### 2. 扩展测试（test1 目录）

```bash
npx vitest -c test1/vitest.config.ts run
```

**依赖配置**：
- 配置文件：`test1/vitest.config.ts`（通过 `-c` 参数指定）
- 测试目录：`test1/**/*.test.ts`
- 覆盖范围：controller + service + entity + dto

**适用场景**：需要更高覆盖率时使用

---

#### 3. 监听模式运行测试（开发时使用）

```bash
pnpm test:watch
```

**依赖配置**：
- 配置文件：`vitest.config.ts`（默认）
- 测试目录：`test/**/*.test.ts`

**适用场景**：日常开发时，持续监听代码变化并自动运行测试

**详细说明**：
当你执行 `pnpm test:watch` 时，Vitest 会进入一种特殊的"监听模式"：

1. **启动监听**：测试运行完成后不会退出，而是持续监听项目中的文件变化

2. **自动重测**：当你修改了代码文件（如 `*.ts`、`*.*.test.ts`）时，Vitest 会自动重新运行受影响的测试

3. **交互界面**：你会看到一个交互式界面，类似这样：
   ```
   press a to run all tests
   press f to run only failed tests
   press q to quit
   ```

4. **只测相关的**：Vitest 非常智能，它会根据你修改的文件，只运行与之相关的测试，而不是运行全部测试

**退出方式**：按 `q` 键退出监听模式

**推荐工作流**：
```bash
# 终端1：运行开发服务器
pnpm dev

# 终端2：运行测试监听模式
pnpm test:watch

# 当你修改代码时，测试会自动运行
# 立即看到测试结果，知道自己的修改是否破坏了功能
```

#### 4. 生成覆盖率报告

##### 基础覆盖率报告

```bash
pnpm test:coverage
```

**依赖配置**：
- 配置文件：`vitest.config.ts`
- 报告输出：`test/test-reports/coverage/`
- 报告格式：text + json + html

**适用场景**：了解代码测试覆盖情况、找出未测试的代码

##### 扩展覆盖率报告

```bash
npx vitest -c test1/vitest.config.ts run --coverage
```

**依赖配置**：
- 配置文件：`test1/vitest.config.ts`
- 报告输出：`test1/test-reports/coverage/`
- 报告格式：text + json + html
- 覆盖范围：controller + service + entity + dto

**详细说明**：
- Vitest 会分析你的代码
- 生成覆盖率报告，显示哪些代码被测试覆盖、哪些没有被覆盖
- 报告类型：文本（终端显示）、JSON、HTML

**报告解读**：
```
| File          | % Stmts | % Branch | % Funcs | % Lines |
|---------------|---------|-----------|----------|---------|
| user.service  |  85.71  |   66.66   |  100     |  85.71  |
| role.service  |  100    |   100     |  100     |  100    |
|---------------|---------|-----------|----------|---------|
| All files     |  92.30  |   80.00   |  100     |  92.30  |
```

- **Stmts（语句覆盖率）**：有多少百分比的语句被执行
- **Branch（分支覆盖率）**：有多少百分比的分支（如 if/else）被执行
- **Funcs（函数覆盖率）**：有多少百分比的函数被调用
- **Lines（行覆盖率）**：有多少百分比的代码行被执行

**推荐**：保持测试覆盖率在 80% 以上

### 测试文件命名规范

- 测试文件必须以 `.test.ts` 结尾
- 测试文件名必须与被测试文件对应：`xxx.service.ts` → `xxx.service.test.ts`

### vitest.config.ts 配置说明

```typescript
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    globals: true,                    // 全局可用 describe/it/expect（无需导入）
    environment: 'node',              // 测试环境：node（不是浏览器）
    include: [                        // 要扫描的测试文件
      'src/**/*.test.ts',            # src 目录下的测试
      'tests/**/*.test.ts'           # tests 目录下的测试
    ],
    coverage: {                      // 覆盖率配置
      provider: 'v8',                // 使用 V8 引擎计算覆盖率
      reporter: ['text', 'json', 'html'],  # 报告格式
    },
  },
});
```

## 编写测试用例

### 测试的基本结构

每个测试文件都遵循以下结构：

```typescript
// 1. 导入测试框架和被测试模块
import { describe, it, expect, vi, beforeEach } from 'vitest';
import { YourService } from './your.service.js';

// 2. 描述测试套件
describe('YourService', () => {
  
  // 3. 声明被测试的服务和 mock 对象
  let yourService: YourService;
  let mockMapper: any;

  // 4. 每个测试前执行的准备代码
  beforeEach(() => {
    // 创建 mock 对象（模拟依赖）
    mockMapper = {
      selectList: vi.fn(),      // 模拟 selectList 方法
      selectById: vi.fn(),      // 模拟 selectById 方法
      insert: vi.fn(),
      updateById: vi.fn(),
      deleteById: vi.fn(),
    };

    // 手动注入 mock（因为没有真实的 IoC 容器）
    yourService = new YourService();
    (yourService as any).yourMapper = mockMapper;
  });

  // 5. 描述具体的测试场景
  describe('方法名', () => {
    
    // 6. 编写测试用例
    it('应该描述测试场景', async () => {
      // 准备（Arrange）：设置 mock 返回值
      mockMapper.selectList.mockResolvedValue([{ id: 1, name: 'test' }]);

      // 执行（Act）：调用被测试的方法
      const result = await yourService.yourMethod();

      // 断言（Assert）：验证结果是否符合预期
      expect(result).toHaveLength(1);
      expect(result[0].name).toBe('test');
    });

  });
});
```

### describe 的作用

`describe` 用于将相关的测试用例组织在一起：

```typescript
describe('UserService', () => {
  // 这些测试都属于 UserService
});

describe('UserService', () => {
  describe('pageUsers', () => {
    // pageUsers 方法的测试
  });
  
  describe('createUser', () => {
    // createUser 方法的测试
  });
});
```

### it 的作用

`it`（也可以用 `test`）用于定义一个具体的测试用例：

```typescript
it('应该返回分页的用户列表', async () => {
  // 测试代码
});

test('用户名不存在时应该抛出异常', async () => {
  // 测试代码
});
```

### beforeEach 的作用

`beforeEach` 在每个测试用例运行前执行，用于设置测试环境：

```typescript
beforeEach(() => {
  // 重置 mock，确保每个测试都是独立的
  vi.clearAllMocks();
  
  // 重新设置 mock
  mockMapper = { ... };
});
```

### expect 常用断言

#### 基本断言

```typescript
// 相等
expect(value).toBe(123);                    // 原始值相等（===）
expect(value).toEqual({ a: 1 });             // 对象深度相等

// 不等
expect(value).not.toBe(null);
expect(value).not.toEqual(other);
```

#### 数组断言

```typescript
// 长度
expect(array).toHaveLength(3);

// 包含
expect(array).toContain(item);
expect(array).toContainEqual({ id: 1 });     // 深度包含

// 开头/结尾
expect(array).toStartWith(item);
expect(array).toEndWith(item);
```

#### 异步断言

```typescript
// Promise 成功
await expect(promise).resolves.toBe('success');

// Promise 失败
await expect(promise).rejects.toThrow('error');
```

#### 异常断言

```typescript
// 期望抛出异常
await expect(service.method()).rejects.toThrow('错误信息');
```

### Mock 函数详解

Mock 函数用于模拟依赖（数据库、API 等），让你专注于测试业务逻辑：

#### 模拟返回值

```typescript
// 模拟返回单个值
mockFn.mockReturnValue(data);

// 模拟返回多个值（依次返回）
mockFn
  .mockReturnValueOnce(第一次返回值)
  .mockReturnValueOnce(第二次返回值)
  .mockReturnValue(之后的默认返回值);

// 模拟异步返回（最常用）
mockFn.mockResolvedValue(data);
mockFn.mockResolvedValueOnce(data);

// 模拟抛出错误
mockFn.mockRejectedValue(new Error('error'));
```

#### 验证调用

```typescript
// 是否被调用
expect(mockFn).toHaveBeenCalled();

// 调用次数
expect(mockFn).toHaveBeenCalledTimes(3);

// 调用参数
expect(mockFn).toHaveBeenCalledWith(arg1, arg2);

// 只调用了一次，且参数正确
expect(mockFn).toHaveBeenCalledOnceWith(arg1, arg2);
```

## Git 提交验证

### 使用 pre-commit hook（推荐）

项目已在 `.git/hooks/pre-commit` 创建了 hook，提交代码时会自动运行测试。

> **注意**：由于是 Windows 环境，需要手动赋予执行权限。Linux/Mac 用户可跳过此步骤。

#### pre-commit hook 是什么？

**pre-commit hook** 是一个 Git 钩子，会在 `git commit` 命令执行后、实际提交代码前自动运行。

工作流程：
```
1. 你执行 git commit -m "修改了什么"
           ↓
2. Git 准备提交代码
           ↓
3. 自动触发 pre-commit hook
           ↓
4. 运行单元测试（pnpm test）
           ↓
5. 测试通过 → 提交成功 ✓
           ↓
   测试失败 → 提交中止 ✗
```

#### Windows 用户设置

```powershell
# 以管理员身份运行 PowerShell
# 赋予执行权限
cmd /c "attrib +x C:\D\project\ai-frist-framework\.git\hooks\pre-commit"
```

#### Linux/Mac 用户设置

```bash
# 赋予执行权限
chmod +x .git/hooks/pre-commit
```

#### 验证 hook 是否生效

```bash
# 尝试提交代码
git commit -m "test"

# 输出类似：
# 🔍 Running unit tests before commit...
# 
#  ✓ src/service/user.service.test.ts (11)
#  ✓ src/service/role.service.test.ts (10)
#  ✓ src/service/menu.service.test.ts (11)
# 
#  ✅ All tests passed!
# 
#  [main xxx] test
#  1 file changed, 10 insertions(+)

# 如果测试失败，输出类似：
# 🔍 Running unit tests before commit...
# 
#  ✗ src/service/user.service.test.ts
# 
#  ❌ Unit tests failed! Commit aborted.
```

### 手动验证测试

如果不想使用 hook，也可以手动运行测试：

```bash
# 方式1：使用 workspace filter
pnpm --filter @scaffold/api test

# 方式2：在目录下运行
cd scaffold/packages/api
pnpm test
```

### 推荐的开发工作流

```bash
# 1. 开发功能
# 2. 修改代码
# 3. 运行测试（快速验证）
pnpm test

# 4. 如果测试通过，提交代码
git add .
git commit -m "feat: 添加新功能"

# 5. hook 会自动再次运行测试
#    如果测试失败，提交不会成功
```

```
| service       |   93.37 |     87.9 |   96.96 |   93.37 |
| utils         |     100 |      100 |     100 |     100 |
|---------------|---------|----------|---------|---------|
```

### 2. 打开 HTML 覆盖率报告

HTML 报告可以直观地看到哪些代码行未被覆盖：

```bash
# 覆盖率报告生成在 test-reports/coverage/ 目录
# 打开 index.html 查看详细报告

# Windows
start test-reports/coverage/index.html

# Mac
open test-reports/coverage/index.html

# Linux
xdg-open test-reports/coverage/index.html
```

HTML 报告中：
- **绿色**：已覆盖的代码
- **红色**：未覆盖的代码
- **黄色**：部分覆盖（如 if 语句只走了 true 分支）

### 3. 分析未覆盖的代码

#### 常见的未覆盖代码类型

| 类型 | 示例 | 说明 |
|------|------|------|
| **错误处理分支** | `catch { throw new Error() }` | 需要模拟抛出异常 |
| **边界条件** | `if (status === 0)` | 需要测试 status=0 的情况 |
| **空值判断** | `if (!user)` | 需要测试用户不存在 |
| **复杂逻辑** | `if (a && b \|\| c)` | 需要覆盖所有分支 |

#### 示例：分析 user.service.ts 的未覆盖行

假设覆盖率报告显示：
```
user.service.ts | 96.47% | 80% | 100% | 96.47% | 149-150,158-161
```

这表示第 149-150 行和 158-161 行未被覆盖。打开 HTML 报告可以看到具体是哪行代码。

### 4. 补充测试用例

#### 步骤1：确定需要补充的测试场景

根据未覆盖的代码，确定需要补充的测试场景：

```typescript
// 原始代码（未覆盖）
async updateUser(id: number, dto: UpdateUserDto) {
  const user = await this.userMapper.selectById(id);
  if (!user) {  // ← 这个分支未覆盖
    throw new Error('用户不存在');
  }
  // ... 更新逻辑
}
```

#### 步骤2：编写测试用例

```typescript
describe('updateUser', () => {
  // 补充：用户不存在的场景

    mockUserMapper.selectById.mockResolvedValue(null);

    await expect(
      userService.updateUser(999, { realName: '新名字' })

  });
```


### 5. 分支覆盖率的优化

分支覆盖率低通常意味着有条件分支未被测试。

#### 示例：提升分支覆盖率

**原始代码**：
```typescript
function validateUser(user: User): boolean {
  if (!user.username || user.username.length < 3) {
    return false;  // 分支1
  }
  if (!user.email || !user.email.includes('@')) {
    return false;  // 分支2
  }
  return true;  // 分支3
}
```

**测试用例**：
```typescript
describe('validateUser', () => {
  it('用户名太短应该返回 false', () => {
    expect(validateUser({ username: 'ab', email: 'test@test.com' })).toBe(false);
  });

  it('邮箱格式错误应该返回 false', () => {
    expect(validateUser({ username: 'abc', email: 'invalid' })).toBe(false);
  });

  it('验证通过应该返回 true', () => {
    expect(validateUser({ username: 'abc', email: 'test@test.com' })).toBe(true);
  });
});
```

### 6. 覆盖率目标建议

| 目标 | 说明 |
|------|------|
| **60%** | 入门级：核心业务逻辑已覆盖 |
| **80%** | 良好：大部分场景已覆盖 |
| **90%** | 优秀：边界条件和异常已覆盖 |
| **100%** | 理想：所有代码都被测试 |

**建议**：
1. 先确保核心 service 层达到 80%+
2. 再补充 controller 层的测试
3. 最后处理边界条件和异常分支

### 7. 使用 it.each 进行参数化测试

当有多个相似场景时，使用参数化测试减少重复代码：

```typescript
// 原始写法：多个相似测试
it('应该处理 status=0', () => { /* ... */ });
it('应该处理 status=1', () => { /* ... */ });
it('应该处理 status=2', () => { /* ... */ });

// 参数化写法
it.each([
  { status: 0, expected: '禁用' },
  { status: 1, expected: '启用' },
  { status: 2, expected: '待审核' },
])('应该处理 status=$status', ({ status, expected }) => {
  expect(getStatusText(status)).toBe(expected);
});
```

## 常见问题

### 1. 测试失败：mock 未正确设置

**问题**：测试报错 `mockFn is not a function`

**原因**：mock 对象没有在 `beforeEach` 中正确设置

**解决**：
```typescript
beforeEach(() => {
  vi.clearAllMocks(); // 清除之前的 mock 状态
  // 重新设置 mock
  mockMapper = {
    selectList: vi.fn(),
    selectById: vi.fn(),
  };
});
```

### 2. 测试通过但实际调用失败

**问题**：测试通过了，但实际运行时功能不正常

**原因**：mock 返回值与实际代码不匹配

**解决**：
```typescript
// 错误：返回 undefined
mockMapper.selectList.mockResolvedValue(undefined);

// 正确：根据实际返回类型设置
mockMapper.selectList.mockResolvedValue([]);  // 返回数组
mockMapper.selectById.mockResolvedValue(null); // 返回 null
```

### 3. 异步测试未完成

**问题**：测试运行完成但没有等待异步操作

**原因**：没有使用 `async/await`

**解决**：
```typescript
// 错误
it('test', () => {
  const result = service.method();
  expect(result).toBeDefined();
});

// 正确
it('test', async () => {
  const result = await service.method();
  expect(result).toBeDefined();
});
```

```bash
it.only('only run this', () => {
  // ...
});
```
